# 需求处理手册(Playbook)

> 给所有团队成员:一个需求下来时,照此执行。Claude 用户直接说触发语(skill 自动响应);Cursor/Codex 用户按各仓库 `prompts/*.md` 执行,流程完全相同。

## 第 0 步:判断需求类型

问一个问题:**这个需求需要改动几个服务的代码?**

- 只改 1 个 → 走【单应用流程】
- 2 个及以上 → 走【跨应用流程】
- 不确定 → 在任意已接入仓库对 AI 说"帮我分析这个需求的影响面"(AI 会读 hub registry 判断);仍拿不准就按跨应用走,总 spec 里影响面只剩一个服务时再降级

判断技巧:凡涉及"调用别的服务的新接口 / 改对外接口 / 新增消息"的,几乎都是跨应用——即使你自己只改一个仓库,契约对端也算影响面。

## 单应用流程(开发者一人闭环)

在该服务仓库内依次执行:

| 步骤 | 你做什么 | 产出 |
|---|---|---|
| 1. 立 spec | `/speckit.specify <需求描述>`(只讲 what/why,不讲技术) | 分支 + `specs/NNN-xxx/spec.md` |
| 2. 澄清 | `/speckit.clarify`,认真回答 AI 的问题(不要跳过,这是减少返工最有效的一步) | spec 补全 Clarifications |
| 3. 定方案 | `/speckit.plan <技术选型与约束>` | plan.md、data-model、contracts |
| 4. 拆任务 | `/speckit.tasks`(复杂需求可加 `/speckit.analyze` 做一致性检查) | tasks.md |
| 5. 实现 | `/speckit.implement`(AI 会先查 `docs/wiki/code-map.md` 定位代码) | 代码 + 测试 |
| 6. 收尾 | 说"需求收尾"(finalize-feature):spec 结论沉淀进 docs/、wiki、ADR;若依赖/契约变了,同步 hub registry | 文档与代码一致 |
| 7. 提 PR | 按 PR 模板勾检查清单;CI 会检查文档新鲜度 | 评审合入 |

## 跨应用流程(发起人协调 + 各服务开发者并行)

### 阶段一:立总 spec(发起人,在 hub——位置见任一应用仓库根的 `.vibe-hub` 文件)

对 AI 说:**"这个需求涉及 X 和 Y 服务:<需求描述>"**(cross-app-spec 自动触发),AI 会:

1. 读 registry 分析影响面,给你确认(存疑的服务会标注)
2. 建 `specs/NNN-需求名/spec.md`:需求概述、影响面表、**契约变更**、各服务职责拆分与验收标准、上线顺序
3. 为每个涉及服务生成一条**拷贝即用的启动指令**

你需要把关的:影响面是否漏了服务、契约设计是否合理、职责边界是否清晰。

### 阶段二:契约评审(人工闸口,不可跳过)

把总 spec 发给涉及服务的 owner 评审,重点审契约章节(接口定义、兼容性标注)。达成一致后把总 spec 状态改为 `contracts-approved`,提交。

**评审通过前,任何服务不得开始实现。** 这一步是跨应用协作质量的关键——契约定稿后各服务才能并行开发互不阻塞。

### 阶段三:各服务并行实现(各开发者,在各自仓库)

每个服务的开发者拿到自己那条启动指令,在自己仓库粘贴执行——之后就是标准【单应用流程】的第 2~7 步(启动指令已预填总 spec 链接、职责摘要、契约约束,子 spec 天然带跨应用上下文)。

各自完成后:回填总 spec 影响面表中自己的子 spec 链接与状态。

### 阶段四:上线与关闭(发起人)

按总 spec 的上线顺序发布(通常先提供方、后消费方,注意兼容性约束);全部上线并通过端到端验收后,总 spec 状态改 `done`;依赖关系有变化的,把 registry 中本需求的 `planned` 依赖转为 `active`(合入后 CI 自动重生成依赖图)。

## 角色速查

| 角色 | 职责 |
|---|---|
| 需求发起人 | 判断类型;跨应用时立总 spec、组织契约评审、盯上线顺序与关闭 |
| 服务 owner | 评审涉及自己服务的契约与 registry 变更 |
| 开发者 | 在自己仓库走 spec-kit 流程 + 收尾;维护本仓库文档 |

## 常见情况

- **做到一半发现要动别的服务**:停下,回 hub 补/升级总 spec(单应用升级为跨应用),补契约评审,再继续。不要"顺手"在对端加接口。
- **紧急线上修复**:可以先修(跳过 spec),但合并后 24h 内补:说"同步文档"(sync-docs)+ 若动了契约,补 registry 与消费方通知。
- **需求做完文档没更新被 CI 警告**:说"需求收尾"或"同步文档",一条命令补齐,不追责。
- **只是重构不改行为**:不需要 spec;改动大时收尾跑一次 rebuild-wiki 防 code-map 失效。
- **合并/rebase 时文档冲突**:先解代码冲突完成合并,然后说"同步文档"(sync-docs)以合并后的代码为准重建文档条目;不要花时间逐行手解文档冲突。长期分支及时回灌 main,减少冲突面(详见 doc-style「分支与文档」)。

## 一图流

```
需求 ──► 改几个服务?
          │
    1 个  │  ≥2 个
    ▼     ▼
 单应用   hub 立总 spec(cross-app-spec:影响面+契约+分工+顺序)
 流程       │
    │     契约评审 ◄── 人工闸口
    │       │ contracts-approved
    │     各服务粘贴启动指令,并行走单应用流程 2~7
    │       │
    ▼       ▼
 specify → clarify → plan → tasks → implement → finalize-feature → PR
                                                  │
                              按上线顺序发布 → 回填总 spec → done
```
